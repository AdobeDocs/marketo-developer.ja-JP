# エラーコード

Marketo REST APIは、HTTP、レスポンス、またはレコードレベルでエラーを返します。 このページでは、各エラータイプについて説明し、関連するエラーコードを一覧表示します。

## 例外の処理とログ

統合で予期しない例外が発生した場合に、リクエストと応答をログに記録します。 期限切れの認証など、一部の例外は、再認証によって処理できます。 その他の例外は、関連するリクエストと応答の詳細をリクエストするサポートからの支援を必要とする場合があります。

## エラータイプ

Marketo REST APIは、通常の操作中に3種類のエラーを返すことができます。

- **HTTP レベル：**&#x200B;が`4xx` コードで示されています。
- **応答レベル：** JSON応答の「errors」配列に含まれています。
- **レコードレベル：** JSON応答の「result」配列に含まれ、「status」フィールドと「reasons」配列で各レコードに示されます。

レスポンスレベルおよびレコードレベルのエラーは、HTTP ステータスコード 200を返します。 すべてのエラータイプについて、HTTP理由フレーズはオプションであり、変更される可能性があるため、評価しないでください。

### HTTP レベルのエラー

通常の操作中に、Marketoは2つのHTTP ステータスコードエラー（`413 Request Entity Too Large`と`414 Request URI Too Long`）を返します。 いずれかのエラーから回復するには、リクエストを変更して再試行してください。 送信前にリクエストサイズを確認することで、これらのエラーを防ぐことができます。

Marketoは、リクエストペイロードが1 MBを超えた場合は413を返し、リードの読み込みでは10 MBを返します。 送信前にリクエストサイズを確認してください。 レコードによってリクエストが制限を超える場合は、それらのレコードを別のリクエストに移動します。

GET リクエストのURIが8 KBを超えると、Marketoは414を返します。 送信前にクエリ文字列の長さを確認してください。 制限を超えた場合は、リクエストメソッドをPOSTに変更し、クエリ文字列をリクエスト本文に配置して、`_method=GET` パラメーターを追加します。 長いURIは、GUIDなどの長いフィルター値を持つ大きなレコードバッチを取得する場合に最も一般的です。

[ID](https://developer.adobe.com/marketo-apis/api/identity/) エンドポイントは、通常、クライアント IDまたはクライアント シークレットが無効であるため、401 Unauthorized エラーを返す可能性があります。 次の表に、HTTP レベルのエラーコードを示します。

#### 応答レベルのエラー

応答が`success` パラメーターをfalseに設定すると、応答レベルのエラーが発生します。 次の構造を使用します。

```json
{
    "requestId": "e42b#14272d07d78",
    "success": false,
    "errors": [
        {
            "code": "601",
            "message": "Unauthorized"
        }
    ]
}
```

「errors」配列内の各オブジェクトには、次の2つのメンバーが含まれます。

- `code`: 601から799までの引用符で囲まれた整数。
- `message`: エラーのプレーンテキストの理由。

6xx コードは、リクエスト全体が失敗し、実行されなかったことを示します。 例えば、新しいアクセストークンをリクエストに再認証して渡すことで、601 「アクセストークンが無効」エラーから回復します。

7xx コードは、データが返されなかったか、要求パラメーターが無効であったために、要求が失敗したことを示します。 原因には、無効な日付または必須パラメーターが欠落していることが含まれます。

### レコードレベル {#record_level_errors}

レコードレベルのエラーは、リクエストは有効でしたが、個々のレコードに対して操作を完了できなかったことを示します。 レコードレベルのエラーを含む応答は、次のパターンに従います。

#### 応答

```json
{
   "requestId":"e42b#14272d07d78",
   "success":true,
   "result":[
      {
         "id":50,
         "status":"created"
      },
      {
         "id":51,
         "status":"created"
      },
      {
         "status":"skipped",
         "reasons":[
            {
               "code":"1005",
               "message":"Lead already exists"
            }
         ]
      }
   ]
}
```

結果配列内のレコードは、リクエスト入力配列内のレコードと同じ順序で表示されます。 各レコードは、ステータスフィールドで示すように、独立して成功または失敗する可能性があります。

失敗したレコードの場合、「ステータス」フィールドは「スキップ」され、レコードには「reasons」配列が含まれます。 各理由には、「コード」メンバーと「メッセージ」メンバーが含まれます。 コードは常に1xxxで、メッセージはレコードがスキップされた理由を説明します。

例えば、「リードを同期」リクエストが「action」を「createOnly」に設定し、送信されたキーの1つにリードが既に存在する場合、応答はコード 1005を返し、「リードは既に存在します」というメッセージを上に示します。
